# AI 활용 사례 등록

사내 AI Usage Dashboard(<http://172.16.11.87:4000/)에> AI 활용 사례를 등록한다.

## 사이트 정보

- URL: `http://172.16.11.87:4000/`
- 로그인: `{이메일}` / `{비밀번호}`
- 사례 등록 API: `POST /api/posts`
- 파일 첨부 API: `POST /api/posts/{postId}/files`

## 실행 흐름

### 1단계: 사례 내용 파악

사용자가 이미 내용을 설명한 경우 그것을 기반으로 작성한다. 정보가 부족하면 다음을 질문한다:

- **무엇을 했는지**: 어떤 업무/문제를 AI로 해결했는지
- **어떤 AI를 썼는지**: Claude, GPT, Gemini, Cursor 중 어떤 것을 사용했는지
- **결과가 어땠는지**: 시간 절감, 자동화, 품질 향상 등 체감 효과
- **결과물이 있는지**: 첨부할 파일이나 링크 (코드, 문서, 스크린샷 등)

현재 대화 맥락에서 답을 유추할 수 있으면 굳이 다시 묻지 않는다.

### 2단계: 필드 채우기

수집한 정보를 바탕으로 아래 필드를 채운다.

| 필드 | API key | 설명 | 필수 |
| --- | --- | --- | --- |
| 제목 | title | 사례 제목 | O |
| 요약 | summary | 사례 내용 상세 | O |
| 카테고리 | category | 아래 목록 중 택1 | O |
| AI 도구 | aiSolution | 복수 선택 가능 | O |
| 효과 태그 | effectTags | 복수 선택 가능 | O |
| 자동화 수준 | automationLevel | 택1 | O |
| 재사용성 | reusability | 택1 | O |
| 결과물 링크 | contentUrl | URL | △ |
| 결과물 파일 | (파일 첨부) | 로컬 파일 경로 | △ |

결과물 파일 또는 링크 중 하나는 반드시 포함해야 한다. 사용자가 결과물을 언급하지 않으면 "첨부할 파일이나 링크가 있으신가요?"라고 물어본다. 대화 중 만든 코드, 문서, 커맨드 파일 등이 있으면 그것을 첨부하도록 제안한다.

#### 카테고리 선택 기준

| 카테고리 | 이런 경우에 선택 |
| --- | --- |
| AI도구 사용팁, 신규 기능 공유 | 도구 사용법, 프롬프트 팁, 새 기능 소개 |
| AI활용한 분석 | 데이터 분석, 리서치, 인사이트 도출 |
| 개발업무를 AI로 해결 | 코딩, 디버깅, 아키텍처 설계, CI/CD |
| 디자인업무를 AI로 해결 | UI/UX, 시안, 디자인 시스템 |
| 업무자동화 AI로 해결 | 반복 작업 자동화, 워크플로우, 봇 |
| AI활용한 문서작업 | 문서 작성, 번역, 요약, 보고서 |
| 다양한 업무를 AI로 해결 | 위 카테고리에 딱 맞지 않는 복합 사례 |

#### AI 도구

Claude, Gemini, GPT, Cursor

#### 효과 태그

자동화, 시간절감, 품질향상, 처리량증가, 창의적기여, 새로운업무시도, 오류감소, 퀄리티향상, 비용절감, 업무몰입도증가, 기타

#### 자동화 수준

| 수준 | 기준 |
| --- | --- |
| AI보조 | 사람이 주도하고 AI가 부분적으로 도움 |
| 반자동화 | 사람이 지시/승인하고 AI가 실행 |
| 완전자동화 | 한 번 설정하면 사람 개입 없이 동작 |

#### 재사용성

| 수준 | 기준 |
| --- | --- |
| 1회성 | 특정 상황에서만 사용 |
| 반복사용 | 본인이 반복적으로 사용 가능 |
| 팀공유 | 팀원들도 사용할 수 있는 형태 |
| 전사배포 | 회사 전체에 배포 가능한 수준 |

### 3단계: 본문(summary) 작성 규칙

본문에는 다음을 구분하여 작성한다:

- `■ 사람이 한 일` — 요구사항 정의, 방향 결정, 검수/승인, 피드백 등
- `■ AI가 한 일` — API 분석, 코드 생성, 데이터 수집, 자동화 구현 등

이렇게 구분하는 이유는 AI가 실제로 어디까지 기여했는지 명확히 전달하기 위해서다. 읽는 사람이 "나도 이렇게 해볼 수 있겠다"고 느끼도록 구체적으로 쓴다.

추가로 `■ 효과` 섹션에서 정량적 효과(시간 절감량, 처리 건수 등)가 있으면 포함한다.

### 4단계: 사용자 확인

채운 필드를 사용자에게 보여주고 승인을 받는다. 절대 확인 없이 바로 등록하지 않는다. 사용자가 수정을 요청하면 반영 후 다시 확인한다.

### 5단계: API 호출

사용자가 승인하면 다음 순서로 실행한다.

**Step 1: 로그인**

```bash
curl -s -c /tmp/ai-dash-cookies.txt "http://172.16.11.87:4000/api/auth/login" \
  -H "Content-Type: application/json" \
  -d '{"email":"{이메일}","password":"{비밀번호}"}'
```

**Step 2: 사례 등록**

```bash
curl -s -b /tmp/ai-dash-cookies.txt "http://172.16.11.87:4000/api/posts" \
  -H "Content-Type: application/json" \
  -X POST \
  -d '{JSON body}'
```

**Step 3: 파일 첨부 (있는 경우)**

파일에 시크릿(토큰, 비밀번호 등)이 포함되어 있으면 마스킹 사본을 만들어서 업로드한다.

```bash
curl -s -b /tmp/ai-dash-cookies.txt "http://172.16.11.87:4000/api/posts/{postId}/files" \
  -X POST \
  -F "files=@{파일경로}"
```

## 에러 처리

| 상황 | 대응 |
| --- | --- |
| 로그인 실패 | 이메일/비밀번호 확인 요청. 초기 비밀번호는 1234 |
| 네트워크 오류 | 사내 네트워크(172.16.x.x) 접속 여부 확인 안내 |
| 등록 실패 (API 에러) | 에러 메시지를 사용자에게 보여주고 원인 안내 |
| 유사 사례 감지 | /api/posts/similar API로 중복 확인 가능. 유사 사례가 있으면 사용자에게 알리고 계속 진행할지 확인 |

## 완료

등록 완료 후 게시글 ID와 결과를 알려준다.
